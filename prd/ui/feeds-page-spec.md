# Feeds 页面规范

> **⚠️ 审查前置要求：在开始执行本文档中的任何设计或开发任务之前，必须先完整审查本文件，并与 `prd/ui-ux-redesign-spec.md` 总纲及 `prd/versions/` 下相关版本文档逐项对照，确认一致性后再开始实施。**

---

## 1. 页面定位

> **Feeds 给控制，不给首页。**

Feeds 不再负责告诉用户今天看什么，而是负责让用户控制系统看什么。

它从"主入口"退回到"数据源管理页"，但不会被隐藏，也不会变成后台配置。

---

## 2. 保留能力

以下能力必须完整保留，不做功能削减：

- 树状 / 分组结构
- 订阅管理（添加/删除/编辑）
- 拖拽排序
- 文件夹管理
- 健康状态
- 同步状态
- OPML 导入/导出

---

### 3.2 列表项信息

| 信息 | 必要性 | 说明 |
|------|--------|------|
| 源名称 | 必须 | 标识 |
| 健康状态 | 必须 | 🟢正常 / 🟡延迟 / 🔴错误 |
| 未读数 | 可选 | 仅在有未读时显示 |
| 同步状态 | 必须 | 最后同步时间 |
| Intelligence 状态 | 未来 | 此源是否经常进入 Today |

---

## 4. 新增能力（未来）

以下能力不在 v2.x 初期实现，但设计时要预留展示位：

### 4.1 Intelligence 轻状态

- 这个源最近是否经常进入 Today
- 这个源是否经常只贡献证据，不贡献主判断
- 这个源是否长期质量低或重复度高

这些信息只作为辅助指标，不改变 Feeds 页"控制层"的主身份。

### 4.2 展示方式

- 以轻量 badge / pill 形式附加在源名称旁
- 不做独立面板
- 不做排行/评分

---

## 5. 视觉特征

Feeds 是三个核心页面中 **最工具化** 的：

- 密度可以更高
- 强调控制感
- 强调可操作性
- 不需要太多留白

视觉等级：**Today 最轻 > Topics 最清 > Feeds 最实**

---

## 6. 交互规则

### 6.1 基本操作

| 操作 | 行为 |
|------|------|
| 点击源 | 展开源的最新文章（或在侧栏展示） |
| 拖拽 | 排序 / 移动分组 |
| 右键 | 上下文菜单（编辑/删除/刷新） |
| [+ Add Feed] | 打开添加订阅对话框 |

### 6.2 分组操作

| 操作 | 行为 |
|------|------|
| 点击分组头 | 展开/折叠分组 |
| 拖拽分组头 | 调整分组顺序 |

---

## 7. 对应版本任务

Feeds 页面的核心功能已存在。重设计的重点是：

- 适配新的三栏布局
- 降低为控制层而非首页
- 预留 Intelligence 轻状态展示位

相关版本：v2.2（导航优先级调整）；三段式布局重构版本待确定（见 `prd/ui/navigation-layout-spec.md` §8 备注）。
