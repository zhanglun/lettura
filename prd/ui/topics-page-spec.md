# Topics 页面规范

> **⚠️ 审查前置要求：在开始执行本文档中的任何设计或开发任务之前，必须先完整审查本文件，并与 `prd/ui-ux-redesign-spec.md` 总纲及 `prd/versions/` 下相关版本文档逐项对照，确认一致性后再开始实施。**

---

## 1. 页面定位

> **Topics 给理解，不给堆叠。**

Topics 是 Today 的延长线，用来回答：

> 这些事情到底是什么、怎么变化、值不值得持续跟踪？

它不是分类页，不是第二个首页，也不是"标签集合"。它是 **理解层**。

### 2.2 列表项信息

每个 Topic 项必须包含：

| 信息 | 必要性 | 说明 |
|------|--------|------|
| Topic 名称 | 必须 | 核心标识 |
| 一句话定义 | 必须 | 解释这个 Topic 是什么 |
| 最近变化提示 | 可选 | "3 new" badge |
| 来源数 / 文章数 | 必须 | 表示覆盖范围 |
| 更新时间 | 必须 | 表示新鲜度 |
| 是否跟踪中 | 必须 | Tracked / Discovered 分区 |

### 2.3 分区

- **Tracked**：用户明确跟踪的 Topic
- **Discovered**：系统发现但用户未跟踪的 Topic

## 3. Topic 详情页

### 3.1 阅读顺序

用户进入 Topic 详情时，必须按以下顺序呈现信息：

1. **这个 Topic 是什么** — 定义 + 一句话概括
2. **最近发生了什么变化** — 时间线 / 变化摘要
3. **哪些来源支持它** — 来源证据列表
4. **如果要深读，先读哪几篇** — 推荐原文

### 3.3 页面组件

| 组件 | 职责 |
|------|------|
| TopicHeader | 名称 + 定义 + 跟踪状态 |
| RecentChanges | 时间线，展示 Topic 的变化脉络 |
| SourceList | 按来源分组的文章证据 |
| StartHere | 推荐深读入口（1-3 篇） |

Topic 的进入、跟踪、取消跟踪、排序、静音、详情页深读入口和空状态，统一见 `prd/ui/topics-interaction-spec.md`。

---

## 5. 目标体验

用户应该在 Topics 页面里感觉：

- 我的关注点被系统整理成了脉络
- 我能看到一个主题的变化轨迹
- 我可以信任这些判断（因为有证据）
- 我能随时深入某个主题
