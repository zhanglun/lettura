# Lettura MVP 设计：Today Intelligence Phase 1

## 1. MVP 目标

MVP 只解决一个问题：

> 用户打开 Lettura 后，能不能在几十秒内知道今天最值得关注的内容，而不是再刷几十篇文章。

这里要验证的不是“AI 多强”，而是：

- 用户是否接受 AI 先替自己做初筛
- 用户是否觉得 Today 页真的节省了时间
- 用户是否愿意基于 Today 结果继续点击原文或深入查看

## 2. MVP 核心范围

Phase 1 只包含四个能力：

### 2.1 Top 5 Key Signals

每天给出最值得关注的少量信息项，而不是文章流。

### 2.2 Why It Matters

每条 signal 都必须说明为什么重要。

### 2.3 来源透明

每条 signal 都必须能展开看到来源文章和来源站点。

### 2.4 基础反馈

用户可以表达：有用 / 不相关 / 继续关注。

## 3. MVP 明确不做

这一阶段不做：

- Topic 页面
- Signal 独立页面
- Ask My Feeds
- 全网搜索
- 复杂 Agent 工作流
- 团队协作
- 多端同步
- 正式长期记忆系统
- 完整 Topic / Event 数据建模

## 4. 目标用户

第一阶段只面向一类最容易验证价值的用户：

- 开发者
- 独立开发者
- 技术内容重度订阅者

原因很简单：这类用户本来就有高质量 RSS / 博客 / GitHub / HN 信息源，也最容易感受到“少刷很多内容”的价值。

## 5. 核心体验

用户每天的最短路径应该是：

1. 打开 Lettura
2. 进入 Today
3. 看到 5 条以内的 Key Signals
4. 快速理解每条为什么重要
5. 只打开 1-2 条需要深入的原文
6. 对部分结果给出简单反馈

也就是说，**Today 不是新的文章列表，而是新的决策入口。**

## 6. 页面结构

Today 页面在 MVP 阶段建议只有三层：

### 6.1 今日概览

一句话总结今天的信息状态，例如：

> 今天 AI Coding、Tauri 和 Browser Agent 相关讨论明显升温。

### 6.2 Key Signals 列表

每条包含：

- 标题
- 一句话结论
- Why It Matters
- 来源数量
- 来源文章入口

### 6.3 反馈区

每条 signal 下提供：

- 有用
- 不相关
- 继续跟踪

如果要继续保持克制，甚至可以先不做 `Worth Reading`、`Low Priority` 这些附加区块，避免首页过早变复杂。

## 7. 内容生成原则

MVP 阶段生成出来的 signal 必须满足四个要求：

- **少**：默认只给 3-5 条
- **清楚**：一句话就能说清主结论
- **可追溯**：必须能看到来源
- **可判断**：用户能迅速决定“值不值得继续看”

如果不能做到这四点，那它就不是 signal，只是换了皮的摘要。

## 8. 成功标准

MVP 成功不是看模型分数，而是看用户行为变化：

- 用户愿意优先看 Today，而不是直接看文章列表
- 用户打开原文的数量下降，但满意度不下降
- 用户能明确说出 “Today 帮我节省了时间”
- 用户开始对结果给出正负反馈，而不是完全忽略

## 9. Phase 1 边界判断

只要出现下面这些迹象，就说明范围开始失控：

- 开始讨论 Topic 关系图和长期知识图谱
- 开始想做 Ask 聊天入口
- 开始试图把所有文章都结构化
- 开始为了未来功能设计过多数据层
- 首页开始堆太多模块，重新变回复杂门户

MVP 的边界应该始终守住一句话：

> 先证明 AI 能不能帮用户每天少刷很多信息，再谈更大的信息组织系统。

## 10. 推荐的下一步

在产品层面，Phase 1 之后只需要回答两个问题：

1. 用户是否真的愿意把 Today 当成默认入口？
2. 用户是否开始信任 Lettura 的“筛选”而不是只把它当成“摘要”？

只有这两个答案是正向的，Phase 2 才值得进入 Topic 层。
