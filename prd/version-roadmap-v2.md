# Lettura 版本路线图

## 1. 版本划分原则

Lettura 的 major / minor 不按单个功能模块切分，而按**用户与产品关系是否发生根本变化**来切分。

判断一个版本是否够格成为 major，至少看三件事：

1. **用户心智是否改变** — 用户来 Lettura 是"刷文章"，还是"看判断"，还是"等提醒"。
2. **核心交互对象是否改变** — 产品围绕的是 `Feed / Article`、`Today / Topic`，还是 `Signal / Ask / Memory`。
3. **产品一句话定位是否改变** — 是 RSS Reader、Intelligence Feed，还是 Personal Intelligence System。

按这个标准，Today 与 Topic 不应拆成两个 major。它们都属于同一次大跃迁：**从 RSS Reader 走向 Personal Intelligence Feed**。

真正的第二个 major，发生在产品从"帮用户筛选和组织信息"升级到"主动发现变化、支持查询和长期认知"时。

---

## 2. 顶层版本结构

### Lettura 1.x

当前产品形态：**传统 RSS Reader**。

- 主交互对象：Feed / Article
- 主用户行为：刷列表、点文章、做阅读管理
- 这是现有基线版本

### Lettura 2.x — Personal Intelligence Feed

这是第一个真正的 major。它代表 Lettura 从"阅读器"转成"判断入口"。

核心变化：

- Feed 从主角降级为数据源
- 首页从文章列表变成 Today 判断入口
- 用户从"自己刷很多"变成"先看系统筛好的结论"
- 再进一步，从 Today 扩展到 Topic，让用户开始按主题理解信息，而不是按 Feed 浏览

### Lettura 3.x — Personal Intelligence Radar / Intelligence OS

这是第二个真正的 major。它代表 Lettura 从"帮你筛和组织"升级到"帮你主动发现变化，并支持持续认知"。

核心变化：

- 从被动查看升级为主动提醒
- 从 Today / Topic 扩展到 Signal
- 引入 Ask、长期记忆、跨时间理解、报告与决策支持
- 用户不只是"知道今天该看什么"，而是开始依赖系统"提醒重要变化、帮助判断"

## 3. Lettura 2.x：Personal Intelligence Feed

### 3.1 版本目标

验证并建立这样一个产品命题：

> 用户愿意把 Lettura 当成默认的信息判断入口，而不是传统 RSS 阅读器。

### 3.2 成功标志

如果 2.x 做得对，用户会开始说两句话：

- **"Today 帮我少刷了很多信息。"**
- **"我开始用它看趋势，而不是只看文章。"**

只要这两句话成立，2.x 就成立。

### 3.3 2.x minor 版本规划

#### 2.1 Today 入口重构

目标：把 Today 从"今日文章过滤页"升级为产品默认入口。

任务：

- 调整首页与导航优先级
- 明确 Today 与 Feed List 的关系
- 重写空状态、首次进入、产品叙事文案

#### 2.2 Top Signals MVP

目标：提供 3–5 条最值得关注的内容，而不是文章流。

任务：

- 定义 Signal 卡片最小字段
- 控制展示数量
- 约束标题与一句话结论长度

#### 2.3 Why It Matters v1

目标：每条 Signal 都回答"为什么值得你看"。

任务：

- 设计 Why 模板
- 要求单条只表达一个主判断
- 避免长摘要伪装成解释

#### 2.4 来源透明

目标：建立用户对 Today 的基本信任。

任务：

- 每条 Signal 支持展开来源文章
- 展示来源站点与数量
- 提供清晰的原文入口

#### 2.5 去重与压缩

目标：解决"同一件事刷很多遍"的问题。

任务：

- 相似文章基础去重
- 合并重复报道
- 低信息密度内容降权

#### 2.6 基础反馈闭环

目标：形成最小用户偏好输入。

任务：

- 有用 / 不相关 / 继续关注
- 记录反馈元数据
- 反馈后的视觉响应与状态变化

#### 2.7 Today 概览

目标：在列表上方给出一句话"今日信息判断"。

任务：

- 定义一句话概览格式
- 控制长度、口吻和抽象度
- 保证与下方 Signals 一致

#### 2.8 深读链路优化

目标：让用户从 Signal 更顺畅地进入深读。

任务：

- 优化展开来源与打开原文顺序
- 明确继续阅读路径
- 降低误点和重复跳转

#### 2.9 Topic 最小引入

目标：从"今日最重要"自然过渡到"这些内容属于同一件事"。

任务：

- 定义 Topic 最小对象
- 建立 Signal → Topic 跳转
- 保持 Today 仍是第一入口

#### 2.10 Topic 基础页

目标：让用户第一次按主题而不是按文章理解信息。

任务：

- Topic 标题与一句话定义
- 来源文章聚合
- 时间范围说明

#### 2.11 Topic 聚合与多观点

目标：让 Topic 不是分类壳，而是真正的理解层。

任务：

- 同 Topic 多来源归组
- 并置不同观点
- Topic-level summary

#### 2.12 Topic 排序与持续跟踪

目标：让 Topic 成为稳定的理解入口。

任务：

- Topic 排序依据
- Continue Tracking 升级
- Topic 使用验证与收口判断

## 4. Lettura 3.x：Personal Intelligence Radar / Intelligence OS

### 4.1 版本目标

建立这样一个新的产品命题：

> Lettura 不只是告诉用户今天该看什么，而是持续发现重要变化，并成为个人认知系统的一部分。

### 4.2 成功标志

如果 3.x 做得对，用户会开始说：

- **"这个产品会提醒我重要变化。"**
- **"我会用它追踪一个领域，而不只是看今天的内容。"**
- **"我可以直接问它，我关心的主题最近发生了什么。"**

### 4.3 3.x minor 版本规划

#### 3.1 Signal 对象定义

目标：把"变化"定义成独立对象。

任务：

- 定义 Signal 边界
- 建立 Topic → Signal 映射
- 分类 Rising / Declining / Unexpected

#### 3.2 Signal 检测 v1

目标：找到值得提醒的变化。

任务：

- 趋势变化规则
- 异常检测最小版
- 时间窗口与置信度约束

#### 3.3 Signal 展示页

目标：让 Signal 成为新的信息入口。

任务：

- Signal 卡片结构
- 变化说明
- 来源与时间维度可视化

#### 3.4 主动提醒机制

目标：从"我来找"升级到"它来提醒"。

任务：

- 提醒触发规则
- 提醒频率控制
- 避免打扰与提醒疲劳

#### 3.5 用户兴趣模型 v1

目标：让提醒开始个性化。

任务：

- 基于订阅、反馈、阅读行为建立兴趣模型
- 高兴趣主题识别
- 低相关内容过滤

#### 3.6 Topic 历史演化

目标：让系统理解"这件事是如何变化过来的"。

任务：

- Topic 时间线
- 关键节点抽取
- 演化摘要

#### 3.7 Ask My Feeds v1

目标：给 Topic / Signal 一个查询入口。

任务：

- 限制问题范围
- 基于用户订阅源作答
- 回答必须附来源

#### 3.8 Ask 深化到研究链路

目标：从问答走向轻量研究辅助。

任务：

- 多来源回答组织
- 补充 why / implications / connections
- 支持后续追问

#### 3.9 长期记忆 v1

目标：让系统开始拥有跨时间理解能力。

任务：

- 用户历史主题记忆
- 阅读行为与偏好沉淀
- 长周期引用能力

#### 3.10 Intelligence Brief / Report

目标：把分散的 Signal 组织成阶段性判断结果。

任务：

- 周报 / 月报
- Topic 级别总结
- 变化与意义结构化表达

#### 3.11 决策支持视图

目标：从"知道发生了什么"走向"帮助判断接下来该关注什么"。

任务：

- implications 展示
- related topics 关联
- next watchpoints 建议

#### 3.12 3.x 收口版

目标：判断 Lettura 是否已经具备"认知基础设施"雏形。

任务：

- Signal 使用验证
- Ask 质量验证
- 长期记忆与报告价值复盘
- 判断是否进入真正的 Intelligence OS 叙事

## 5. 版本拆分控制原则

为了保证每个 minor 都是可迭代、可交付、可验证的小步更新，后续必须坚持以下几个约束。

### 5.1 每个 minor 只解决一个主问题

不要在一个 minor 里同时做"体验升级 + 数据层重构 + AI 能力拓展 + 交互重写"。

### 5.2 每个 minor 只有一个主收益

一个 minor 要么提升信任，要么提升理解，要么提升提醒能力，不要混成综合包。

### 5.3 每个 minor 只有一个主验收点

避免一个版本包含太多模糊目标。例如：

- 2.4 的验收点是"来源透明是否增强信任"
- 3.4 的验收点是"主动提醒是否被认为有价值而非打扰"

### 5.4 基础建设嵌入 minor，而不是单独拉出巨大基建版本

除非出现明显架构门槛，否则不要做"纯基建大版本"。应把必要的数据层、质量治理、模型调优嵌入具体 minor 中完成。

### 5.5 始终服从产品哲学

每个版本都必须回答三个问题：

1. 是否让用户**看得更少**？
2. 是否让用户**理解更深**？
3. 是否让用户**判断更快**？

只要有版本回答不了这三件事，就不该进入路线图。

---

## 6. 总结

最终建议的版本路线图如下：

- **1.x = RSS Reader**
- **2.x = Personal Intelligence Feed**（Today + Topic 都在这个 major 内演进）
- **3.x = Personal Intelligence Radar / Intelligence OS**（Signal + Ask + 长期记忆 + 报告 + 决策支持）

这个划分的核心不是功能数量，而是**产品范式变化**：

- **1.x → 2.x**：从"刷文章"到"看判断"
- **2.x → 3.x**：从"看判断"到"收提醒 / 做追踪 / 形成认知"
