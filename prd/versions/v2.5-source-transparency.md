## §1 需求摘要

**来源**：`prd/version-roadmap-v2.md` §3.3 · 2.5

**目标**：建立用户对 Today 的基本信任。

**功能范围**：
- 每条 Signal 支持展开来源文章列表
- 展示来源站点名称与数量
- 提供清晰的原文入口（点击跳转到 ArticleReader）
- 来源文章按时间排序，显示标题、来源 Feed、发布时间
- 可选：50 字摘录预览

**明确不做**：
- ❌ 去重与压缩（2.6）
- ❌ 用户反馈闭环（2.7）
- ❌ Today Overview（2.8）
- ❌ Topic 对象（2.9+）
- ❌ 完整 Signal 详情页（只做卡片内展开）

**验收标准**：
1. 每张 Signal 卡片底部有"展开来源"按钮
2. 点击展开后显示该 Signal 的所有来源文章
3. 每条来源文章显示：标题、Feed 名称、发布时间
4. 点击来源文章可打开现有 ArticleReader 阅读原文
5. 展开动画流畅（300ms ease-in-out）

**成功指标**：
- 来源展开率 > 60%（用户愿意查看来源）
- 从来源点击进入阅读的比例 > 30%
- 用户信任感提升（定性反馈）

## §2 技术方案

### 2.1 数据库变更

**无新表 / 无新迁移**。来源文章数据来源于 `article_ai_analysis` 关联的 `articles` 表，通过 Signal 的 `id`（即 `article_ai_analysis.id`）查询相关文章。

### 2.2 Rust 后端

#### 2.2.1 新增 IPC 命令

**新增**：`get_signal_detail`

```rust
// cmd.rs
#[command]
pub async fn get_signal_detail(
    signal_id: i32,
    state: State<'_, AppState>,
) -> Result<SignalDetail, String> {
    // 1. 查询 article_ai_analysis WHERE id = signal_id
    // 2. 查询该 Signal 的来源文章
    //    方案 A：聚类组内所有文章（通过 topic_id 或 embedding 相似度匹配）
    //    方案 B：直接返回该 article_id 对应的单篇文章
    //    推荐方案 A（聚类组），因为一个 Signal 代表一组相关文章
    // 3. 返回 SignalDetail
}
```

**SignalDetail 返回结构**：

```rust
pub struct SignalDetail {
    pub signal: Signal,                // 同 v2.3 的 Signal 结构
    pub all_sources: Vec<SignalSource>, // 完整来源列表（不截断）
    pub related_signals: Option<Vec<Signal>>, // 相关 Signal（可选，预留）
    pub topic_context: Option<TopicBrief>,    // 所属 Topic（可选，2.9+ 填充）
}

pub struct TopicBrief {
    pub id: i32,
    pub title: String,
    pub description: String,
    pub article_count: i32,
    pub status: String,
}
```

> **实现说明**：v2.3 的 `get_today_signals` 已返回每条 Signal 的 `sources: Vec<SignalSource>`，但可能截断（最多 10 条）。`get_signal_detail` 返回完整列表，适用于"查看全部"场景。

**Signal 来源文章查询逻辑**：

> **现状说明**：`article_ai_analysis.topic_id` 列存在但始终为 NULL（v2.3 pipeline 的 `simple_cluster` 结果未持久化到该列）。因此 v2.5 采用 **embedding 相似度匹配**方案查找来源文章。

```rust
// ai/pipeline.rs 或 cmd.rs 中实现
fn find_related_sources(
    conn: &mut SqliteConnection,
    signal_id: i32,
    similarity_threshold: f64,  // 默认 0.7（比 dedup 的 0.85 宽松）
) -> Result<Vec<SignalSource>, String> {
    // 1. 查询目标 Signal 的 embedding_json
    //    SELECT embedding_json FROM article_ai_analysis WHERE id = :signal_id
    //    解析为 Vec<f32>
    //
    // 2. 查询所有有 embedding 的文章
    //    SELECT ai.article_id, ai.embedding_json, a.title, a.link, a.feed_uuid, a.pub_date
    //    FROM article_ai_analysis ai
    //    JOIN articles a ON ai.article_id = a.id
    //    WHERE ai.embedding_json IS NOT NULL
    //      AND ai.id != :signal_id
    //
    // 3. 逐一计算 cosine similarity
    //    similarity > threshold 的文章视为来源
    //
    // 4. 加上目标 Signal 自身的 article_id（主文章）
    //
    // 5. 按 pub_date DESC 排序，组装 Vec<SignalSource>
}
```

**降级策略**：
- 如果目标 Signal 无 embedding（`embedding_json IS NULL`）→ 仅返回该 Signal 的主文章（单条来源）
- 如果无相似文章（所有 similarity < threshold）→ 仅返回主文章
- 查询结果限制最多 50 条来源文章（防止大量相似文章拖慢前端）

#### 2.3.2 组件变更

**修改**：`layout/Intelligence/SignalCard.tsx`

在卡片底部新增"展开来源"按钮：

```
SignalCard（v2.5 扩展）:
├── 标题
├── Summary
├── 💡 Why（v2.4）
├── 📰 来源计数 "5 articles · 3 sources"     ← 已有，v2.3
```

**交互流程**：

1. 用户看到 Signal 卡片，底部显示 "📰 5 articles · 3 sources"
2. 点击"展开来源"按钮或来源计数行
3. SignalSourceList 向下滑出（300ms ease-in-out）
4. 默认显示前 5 条来源文章
5. 如果超过 5 条，显示"显示全部 N 篇 ▼"
6. 点击"显示全部"调用 `get_signal_detail` 获取完整列表
7. 点击某条来源文章 → 打开现有 ArticleReader（复用现有路由 `/local/feeds/:uuid/articles/:id`）
