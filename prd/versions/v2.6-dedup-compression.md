## §1 需求摘要

**来源**：`prd/version-roadmap-v2.md` §3.3 · 2.6

**目标**：解决"同一件事刷很多遍"的问题——当多个来源报道同一事件时，Today 页面只展示一条聚合后的 Signal，而非多条重复内容。

**功能范围**：
- 基于 Embedding 向量的相似文章检测（cosine similarity > 0.85 视为重复候选，该阈值需在开发阶段调优）
- 重复报道合并：保留质量最高的源文章为主 Signal，其余标记为"相关"并附加在来源列表
- 低信息密度内容降权：通过规则（文章内容过短、无实质事实陈述）或 LLM 判断降低 relevance_score
- Pipeline 新增去重步骤：在 embedding 之后、聚类之前，先做相似度检查，命中阈值的文章标记为 duplicate

**明确不做**：
- ❌ 跨天去重（本版本只处理当日文章）
- ❌ 用户自定义去重阈值
- ❌ 去重结果的批量管理 UI（Settings 展示去重统计即可）
- ❌ 语义层面的"深层去重"（如不同角度报道同一事件仍保留）

**验收标准**：同一事件被 5+ 个源报道时，Today 页面只出现 1 条 Signal，该 Signal 的来源列表包含全部相关文章。低信息密度文章（< 100 字正文）的 relevance_score 降低 0.3+。

**成功指标**：
- 重复 Signal 出现率 < 10%（人工评审 3 天数据）
- 去重步骤不增加 Pipeline 总耗时 > 20%
- 无误合并（不同事件被错误归为重复）率 < 2%

fn find_duplicate_groups(

/// 评估单篇文章的信息密度

**Pipeline 变更**（`ai/pipeline.rs`）：

在 `Embed → StoreVec` 之后、`Cluster` 之前，插入去重步骤：

```
原流程: Fetch → Embed → StoreVec → Cluster → LLM → Rank → Store
新流程: Fetch → Embed → StoreVec → Dedup → Cluster → LLM → Rank → Store
                                          ↑ 新增步骤
```

去重步骤的具体行为：
1. 从 sqlite-vec 取出本次 batch 所有新文章的 embedding
2. 对每对文章计算 cosine similarity
3. similarity > 0.85 的文章组 → 选质量最高（`sources.quality_score` 最高）为主文章
4. 其余标记 `is_duplicate = true`，`duplicate_of = primary_id`
5. 主文章的 `source_count` 合并被合并文章的来源数

**信息密度降权逻辑**：

```rust
fn apply_density_penalty(base_score: f32, density: &DensityResult) -> f32 {
    if density.is_low_density {
        (base_score - 0.3).max(0.0)
    } else if density.density < 0.5 {
        (base_score - 0.1).max(0.0)
    } else {
        base_score
    }
}
```

### 2.4 依赖

**新增 Rust 依赖**：sqlite-vec（通过 rusqlite extension 加载）

```toml
# apps/desktop/src-tauri/Cargo.toml
[dependencies]
# sqlite-vec 通过 rusqlite 的 load_extension 加载
# 不直接添加 crate，而是编译 sqlite-vec C 库并在运行时加载
```

sqlite-vec 集成方式：
1. 从 https://github.com/asg017/sqlite-vec 获取预编译的 `vec0.so`（macOS）/ `vec0.dll`（Windows）
2. 随应用打包，运行时通过 `rusqlite::Connection::load_extension` 加载
3. 使用 `vec0` 虚拟表存储 embedding 向量
