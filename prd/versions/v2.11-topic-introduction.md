## §1 需求摘要

**来源**：`prd/version-roadmap-v2.md` §3.3 · 2.11

**目标**：从"今日最重要"自然过渡到"这些内容属于同一件事"。

**前置版本状态**：
- v2.10 已完成三段式布局迁移：Rail 中已包含 Topics 图标（`Layers`），路由 `/local/topics` 已注册，`TopicsPage` 占位空状态页面已存在于 `src/pages/TopicsPage.tsx`
- v2.6 的 `dedup.rs` 已实现 cos > 0.85 的重复检测，pipeline 中 `simple_cluster` 已实现 cos > 0.7 的聚类
- `article_ai_analysis` 表已有 `topic_id` 列（Nullable<Integer>），但未被使用
- `uuid` crate 已在 Cargo.toml 中

**功能范围**：
- 定义 Topic 最小对象（`topics` 表 + `topic_articles` 关联表）
- Pipeline 聚类步骤：当非重复文章形成 cluster（≥2 篇）时自动创建/更新 Topic，写入 `topic_articles`
- 实际使用 `article_ai_analysis.topic_id` 字段，将 Signal 关联到 Topic
- SignalCard 组件增加 Topic badge：若 Signal 关联了 Topic，显示可点击的 Topic badge
- 升级 `TopicsPage`（v2.10 占位）为数据驱动页面，调用 `get_topics` IPC 展示 Topic 列表
- 新增 2 个 IPC 命令：`get_topics` / `get_topic_detail`
- 改造 `get_today_signals` IPC：SQL JOIN topics 表填充 `topic_id` 和 `topic_title`

**明确不做**：
- ❌ Topic 详情页完整组件（v2.12）
- ❌ Topic 排序 / 关注 / 跟踪功能（v2.12–2.13）
- ❌ Topic-level summary 生成（v2.12）
- ❌ 新增前端 Zustand slice（直接在组件内 useState + useEffect）
- ❌ `article_ai_analysis.topic_id` 添加 REFERENCES 约束（保持裸 INTEGER，避免迁移复杂度）

**验收标准**：Pipeline 运行后，聚类产生的文章组自动形成 Topic，SignalCard 上显示 Topic badge，点击 badge 或 Rail Topics 图标可到达 `/local/topics` 页面并看到 Topic 列表。

**前置版本**：2.6（向量去重/聚类基础）、2.10（三段式布局 + Topics 路由 + TopicsPage 占位）

**成功指标**：
- Pipeline 运行后，`topics` 表有记录，`topic_articles` 关联正确
- SignalCard 上有 `topic_id` 的卡片显示 Topic badge
- Topic badge 点击后跳转到 `/local/topics`，列表可正常展示
- Rail Topics 图标点击也能到达同一页面

## §2 技术方案

### 2.1 数据库变更

**新增 `topics` 表 + `topic_articles` 表**（迁移 `2026-07-01-170000_create_topics_tables`）：

> **注意**：迁移编号必须大于现有最新迁移 `2026-06-01-160000_create_user_feedback`。

```sql
-- up.sql
CREATE TABLE topics (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    uuid TEXT NOT NULL UNIQUE,
    title TEXT NOT NULL,
    description TEXT,
    status TEXT NOT NULL DEFAULT 'active' CHECK(status IN ('active', 'archived', 'merged')),
    article_count INTEGER NOT NULL DEFAULT 0,
    source_count INTEGER NOT NULL DEFAULT 0,
    first_seen_at TIMESTAMP,
    last_updated_at TIMESTAMP,
    create_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_topics_status ON topics(status);
CREATE INDEX idx_topics_updated ON topics(last_updated_at);

CREATE TABLE topic_articles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    topic_id INTEGER NOT NULL REFERENCES topics(id) ON DELETE CASCADE,
    article_id INTEGER NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
    relevance_score REAL,
    create_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(topic_id, article_id)
);

CREATE INDEX idx_topic_articles_topic ON topic_articles(topic_id);
CREATE INDEX idx_topic_articles_article ON topic_articles(article_id);
```

```sql
-- down.sql
DROP TABLE IF EXISTS topic_articles;
DROP TABLE IF EXISTS topics;
```

**`article_ai_analysis` 表已有 `topic_id` 字段**（v2.3 迁移中已定义为 `Nullable<Integer>`，无 REFERENCES 约束），本版本实际使用该字段。保持现有裸 INTEGER 不变，避免 ALTER TABLE 复杂度。

**新增 IPC 命令**：

| 命令 | 签名 | 说明 |
|------|------|------|
| `get_topics` | `fn(status?: string, sort?: string, limit?: number) -> Result<Vec<TopicItem>, String>` | 返回 Topic 列表 |
| `get_topic_detail` | `fn(topic_id: number) -> Result<TopicDetail, String>` | 返回 Topic 详情 + 关联文章 |

**TopicItem 返回结构**：

```typescript
interface TopicItem {
  id: number;
  uuid: string;
  title: string;
  description: string | null;
  status: "active" | "archived" | "merged";
  article_count: number;
  source_count: number;
  first_seen_at: string | null;
  last_updated_at: string | null;
  is_following: boolean;  // v2.11 暂固定返回 false
}
```

**TopicDetail 返回结构**：

```typescript
interface TopicDetail {
  id: number;
  uuid: string;
  title: string;
  description: string | null;
  status: "active" | "archived" | "merged";
  article_count: number;
  source_count: number;
  first_seen_at: string | null;
  last_updated_at: string | null;
  is_following: boolean;          // v2.11 暂固定返回 false
  recent_changes: string | null;  // v2.11 暂返回 null
  articles: TopicArticle[];
}

interface TopicArticle {
  article_id: number;
  title: string;
  link: string;
  feed_title: string;
  pub_date: string;
  relevance_score: number;
  excerpt: string | null;
}
```

**Pipeline 聚类步骤增强**（`ai/pipeline.rs`）：

```rust
// 伪代码：聚类后的 Topic 处理
fn assign_or_create_topic(conn: &mut SqliteConnection, cluster: &ArticleCluster) -> Result<i32> {
    // 仅对 cluster.len() >= 2 的非重复文章执行
    // 1. 尝试匹配已有 Topic（基于 centroid 距离）
    // 2. 距离 > 阈值 → 创建新 Topic（INSERT topics，生成 uuid + title）
    // 3. 距离 ≤ 阈值 → 更新已有 Topic（UPDATE article_count, source_count, last_updated_at）
    // 4. INSERT topic_articles（每篇文章关联到 Topic，ON CONFLICT 跳过）
    // 5. UPDATE article_ai_analysis SET topic_id = ? WHERE article_id IN (...)
    // 6. 返回 topic_id
}
```

**Topic 标题生成策略**：初版使用 cluster 中 `pub_date` 最新文章的标题作为 Topic title（避免强依赖 LLM）。`description` 字段可选调用 LLM 生成（有 API key 时生成，无 API key 时为 null）。

**与 dedup 的关系**：`dedup.rs` 在 cos > 0.85 时标记重复（同一事件的重复报道），Topic 聚类在 cos > 0.7 时归为同一话题（相关但不同的报道）。Pipeline 执行顺序：先 dedup（标记 `is_duplicate = true`），再 cluster（仅对 `is_duplicate = false` 的文章做 topic 聚类）。

**错误码**：`TOPIC_NOT_FOUND`、`TOPIC_EMPTY`

**`get_today_signals` 改造**（`cmd.rs`）：

当前 `get_today_signals` 硬编码 `topic_id: None, topic_title: None`。改造为：
1. SQL LEFT JOIN topics 表：`LEFT JOIN topics ON article_ai_analysis.topic_id = topics.id`
2. SELECT topics.title AS topic_title
3. 填充 Signal 的 `topic_id` 和 `topic_title` 字段

```tsx
// 使用 useState + useEffect 调用 IPC（不引入独立 Zustand slice）
const [topics, setTopics] = useState<TopicItem[]>([]);
const [loading, setLoading] = useState(true);

useEffect(() => {
  getTopics({ status: 'active', sort: 'recent', limit: 20 })
    .then(setTopics)
    .finally(() => setLoading(false));
}, []);
```
