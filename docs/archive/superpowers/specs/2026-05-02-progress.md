## Phase 6: Today / Topics / Feeds 更新机制

### 背景问题

Today、Topics、Feeds 目前主要依赖页面进入时加载，缺少统一更新机制。用户无法明确知道“订阅源是否已经同步、Today 是否已重新分析、Topics 是否根据新文章更新”。按照原产品设定，这三个页面应该共享同一条内容更新管线。

### 产品目标

- Feeds 是内容入口：负责同步订阅源、记录 feed health、last sync、error。
- Today 是每日判断入口：基于新文章触发 daily intelligence 分析，支持手动重新生成和增量分析。
- Topics 是长期追踪层：由 Today signals、新文章分析结果和 follow 状态增量更新 topic evidence、trend、last_updated。
- 三个页面共享同一个更新状态，而不是各自做孤立 refresh。

### 推荐设计

引入统一 `Refresh Center` / `Sync Status` 状态模型：

- `idle`
- `syncing_feeds`
- `analyzing_today`
- `updating_topics`
- `completed`
- `failed`

状态需要包含：

- `phase`
- `started_at`
- `finished_at`
- `progress`
- `last_success_at`
- `error`
- `updated_counts`：feeds/articles/signals/topics

### 实施计划

- [ ] 审查现有 scheduler、feed refresh、daily intelligence pipeline、topic update 相关代码，确认已有后端入口。
- [ ] 后端补统一 refresh endpoint，例如 `POST /api/refresh/run`，支持参数：
  - `scope=feeds|today|topics|all`
  - `mode=incremental|full`
- [ ] 后端补 refresh status endpoint，例如 `GET /api/refresh/status`。
- [ ] 前端 store 增加 refresh slice，维护 refresh status、triggerRefresh、pollStatus。
- [ ] Feeds 页面增加“刷新订阅源”入口和 last sync / health / error 展示。
- [ ] Today 页面增加“重新生成今日摘要”和“分析新文章”入口；刷新中展示统一 progress。
- [ ] Topics 页面进入时不只 `fetchTopics`，还要显示 last_updated，并在 refresh 完成后重新拉取 topics。
- [ ] 失败时使用 `showErrorToast`，页面内保留可重试入口。
- [ ] 增加测试：
  - refresh slice 状态流转
  - Today 点击刷新触发正确 scope/mode
  - Topics refresh 完成后重新加载
  - Feeds refresh 失败显示错误

### 验收标准

- 用户在 Today、Topics、Feeds 任一页面触发刷新，都能看到一致的状态反馈。
- 同步 feeds 后，Today 能分析新增文章，Topics 能更新相关 evidence 和 last_updated。
- 页面切换不会丢失 refresh 状态。
- 后端失败不会静默失败，用户能重试。

## Phase 7: 未读筛选下点击文章后列表消失 ✅

### 产品目标

- 点击未读文章后，详情应该稳定打开。
- 当前列表中被打开的文章不要立刻消失。
- 已读状态仍然应该写入后端，但列表移除应该延迟到用户主动刷新、切换筛选、离开页面或关闭详情之后。

- [x] 视觉反馈已有：`ArticleItem` 根据本地 `readStatus` state 显示灰色文字 + 移除蓝点
- [x] 切换筛选、手动刷新、翻页时 SWR revalidate 会按后端真实状态重新加载
- [x] 测试：`Article/__tests__/UnreadFilterRetention.test.tsx`（10 tests）

  - 点击已读文章不触发 onRead ✅
  - 已读文章仍触发 updateArticleStatus ✅
  - `retainArticleAfterRead` 纯函数：pages.list 长度不变 ✅
  - `retainArticleAfterRead`：目标 article 仍在列表、read_status=READ ✅

## Phase 8: Today Focus Source Articles + Topic 收敛

### 背景问题

Today Focus 中点击 `Source Articles` 后，右侧面板会出现，但文章内容没有显示；并且只有点击 `Source Articles` 才能激活右侧区域。展开后如果不主动关闭右侧详情，切换另一个 Today source articles 会受阻。当前状态像是 `source list 展开`、`active source`、`article detail` 三件事绑在一起。

同时，本轮 Today/Topic 审查还发现：

- SignalCard 的 Topic 标签只进入 Topics 列表，不能进入对应 Topic detail。
- Today feedback 按钮带 emoji，不符合 PRD 的纯文字轻量控制要求。
- Topic detail 缺少 PRD 要求的 Recent Changes 和 Start Here。
- Topics 空状态保留了重复硬编码中文和英文单位，存在 i18n 噪音。

### 产品目标

- `Source Articles` 按钮只负责展开/收起来源列表。
- 点击任意 source article 都能打开右侧文章详情。
- 点击另一个 signal/source article 时，右侧详情应直接切换，不要求用户先关闭。
- 右侧详情要有明确 loading / error / empty / content 状态。
- 文章内容必须能显示；如果 source 只有 uuid，需要主动 fetch article detail。
- Topic 标签必须进入对应 Topic detail，而不是只进入 Topics 列表。
- Feedback 按钮使用纯文字：有用 / 不相关 / 继续关注。
- Topic detail 至少补齐最小可用的 Recent Changes 和 Start Here。
- Topics 空状态不出现重复文案、硬编码中文或未 i18n 的 `articles/sources` 单位。

### 推荐设计

把 Today source 交互拆成独立状态：

- `expandedSignalId`：哪个 signal 的 source list 展开。
- `activeSourceArticleUuid` 或 `activeSourceArticleId`：右侧正在显示哪篇 source article。
- `sourceArticleDetail`：右侧详情实际加载到的 article 内容。
- `sourceArticleLoading`：右侧文章详情加载中。
- `sourceArticleError`：右侧文章详情加载失败。

交互规则：

- 点击 `Source Articles`：只切换 `expandedSignalId`。
- 点击 source item：设置 `activeSourceArticleUuid`，打开/替换右侧详情。
- 点击另一个 source item：直接替换右侧详情。
- 点击另一个 signal 的 source item：更新 active source，并自动切到新详情。
- 关闭右侧详情：只清空 `activeSourceArticleUuid/sourceArticleDetail`，不强制收起 source list。
- source item 主点击只打开右侧详情；打开完整 Article View / 原文链接作为二级动作。
- Today inline source 阅读 **不自动修改 `read_status`**，避免影响 Feed unread 行为；只有进入完整 Article View 或用户显式操作时才按原有逻辑标记已读。
- 文章正文优先复用现有 Article `View` 的渲染能力，或使用既有安全 HTML 渲染方式；不要把 HTML content 当纯文本直接拆段。
